## <a name="problem-statement"></a><span data-ttu-id="ffb83-109">問題の説明</span><span class="sxs-lookup"><span data-stu-id="ffb83-109">Problem statement</span></span>

<span data-ttu-id="ffb83-110">全体的な問題の説明は、1つの文で簡潔に記述できます。後で取得するために信頼できる情報を保持する必要がありますが、永続化メカニズムを信頼していません。</span><span class="sxs-lookup"><span data-stu-id="ffb83-110">The overall problem statement can be succinctly stated in a single sentence: I need to persist trusted information for later retrieval, but I don't trust the persistence mechanism.</span></span> <span data-ttu-id="ffb83-111">Web 用語では、"信頼された状態を信頼されていないクライアント経由でラウンドトリップする必要がある" として記述されている可能性があります。</span><span class="sxs-lookup"><span data-stu-id="ffb83-111">In web terms, this might be written as "I need to round-trip trusted state via an untrusted client."</span></span>

<span data-ttu-id="ffb83-112">この例では、認証 cookie またはベアラートークンが使用されています。</span><span class="sxs-lookup"><span data-stu-id="ffb83-112">The canonical example of this is an authentication cookie or bearer token.</span></span> <span data-ttu-id="ffb83-113">サーバーは "I am Groot と" xyz アクセス許可 "トークンを生成し、クライアントに渡します。</span><span class="sxs-lookup"><span data-stu-id="ffb83-113">The server generates an "I am Groot and have xyz permissions" token and hands it to the client.</span></span> <span data-ttu-id="ffb83-114">将来の日付では、クライアントはそのトークンをサーバーに返しますが、サーバーでは、クライアントがトークンを偽造していないことを保証する必要があります。</span><span class="sxs-lookup"><span data-stu-id="ffb83-114">At some future date the client will present that token back to the server, but the server needs some kind of assurance that the client hasn't forged the token.</span></span> <span data-ttu-id="ffb83-115">そのため、最初の要件: 信頼性 (</span><span class="sxs-lookup"><span data-stu-id="ffb83-115">Thus the first requirement: authenticity (a.k.a.</span></span> <span data-ttu-id="ffb83-116">整合性、改ざん防止)。</span><span class="sxs-lookup"><span data-stu-id="ffb83-116">integrity, tamper-proofing).</span></span>

<span data-ttu-id="ffb83-117">永続化された状態はサーバーによって信頼されているため、この状態には、オペレーティング環境に固有の情報が含まれている可能性があります。</span><span class="sxs-lookup"><span data-stu-id="ffb83-117">Since the persisted state is trusted by the server, we anticipate that this state might contain information that's specific to the operating environment.</span></span> <span data-ttu-id="ffb83-118">ファイルパス、アクセス許可、ハンドルまたはその他の間接参照、またはサーバー固有のその他のデータの形式を使用できます。</span><span class="sxs-lookup"><span data-stu-id="ffb83-118">This could be in the form of a file path, a permission, a handle or other indirect reference, or some other piece of server-specific data.</span></span> <span data-ttu-id="ffb83-119">通常、このような情報は、信頼されていないクライアントに公開することはできません。</span><span class="sxs-lookup"><span data-stu-id="ffb83-119">Such information should generally not be disclosed to an untrusted client.</span></span> <span data-ttu-id="ffb83-120">そのため、2番目の要件は機密性です。</span><span class="sxs-lookup"><span data-stu-id="ffb83-120">Thus the second requirement: confidentiality.</span></span>

<span data-ttu-id="ffb83-121">最後に、最新のアプリケーションがコンポーネント化されているため、システム内の他のコンポーネントに関係なく、個々のコンポーネントでこのシステムを利用することが求められています。</span><span class="sxs-lookup"><span data-stu-id="ffb83-121">Finally, since modern applications are componentized, what we've seen is that individual components will want to take advantage of this system without regard to other components in the system.</span></span> <span data-ttu-id="ffb83-122">たとえば、ベアラートークンコンポーネントがこのスタックを使用している場合は、同じスタックを使用している可能性のある CSRF メカニズムからの干渉なしで動作する必要があります。</span><span class="sxs-lookup"><span data-stu-id="ffb83-122">For instance, if a bearer token component is using this stack, it should operate without interference from an anti-CSRF mechanism that might also be using the same stack.</span></span> <span data-ttu-id="ffb83-123">そのため、最終的な要件は分離です。</span><span class="sxs-lookup"><span data-stu-id="ffb83-123">Thus the final requirement: isolation.</span></span>

<span data-ttu-id="ffb83-124">要件の範囲を絞るために、さらに制約を与えることができます。</span><span class="sxs-lookup"><span data-stu-id="ffb83-124">We can provide further constraints in order to narrow the scope of our requirements.</span></span> <span data-ttu-id="ffb83-125">ここでは、cryptosystem 内で動作するすべてのサービスが同じように信頼されており、直接制御しているサービスの外部でデータを生成または使用する必要がないことを想定しています。</span><span class="sxs-lookup"><span data-stu-id="ffb83-125">We assume that all services operating within the cryptosystem are equally trusted and that the data doesn't need to be generated or consumed outside of the services under our direct control.</span></span> <span data-ttu-id="ffb83-126">さらに、web サービスに対する各要求が cryptosystem を1回以上通過する可能性があるため、可能な限り高速操作を行う必要があります。</span><span class="sxs-lookup"><span data-stu-id="ffb83-126">Furthermore, we require that operations are as fast as possible since each request to the web service might go through the cryptosystem one or more times.</span></span> <span data-ttu-id="ffb83-127">これにより、対称暗号化がシナリオに適したものになるため、必要な時間まで非対称暗号化を割引できます。</span><span class="sxs-lookup"><span data-stu-id="ffb83-127">This makes symmetric cryptography ideal for our scenario, and we can discount asymmetric cryptography until such a time that it's needed.</span></span>

## <a name="design-philosophy"></a><span data-ttu-id="ffb83-128">設計思想</span><span class="sxs-lookup"><span data-stu-id="ffb83-128">Design philosophy</span></span>

<span data-ttu-id="ffb83-129">まず、既存のスタックに関する問題を特定しました。</span><span class="sxs-lookup"><span data-stu-id="ffb83-129">We started by identifying problems with the existing stack.</span></span> <span data-ttu-id="ffb83-130">これを行った後は、既存のソリューションのランドスケープを調査し、既存のソリューションには、お探しの機能がまったくないという結論を与えました。</span><span class="sxs-lookup"><span data-stu-id="ffb83-130">Once we had that, we surveyed the landscape of existing solutions and concluded that no existing solution quite had the capabilities we sought.</span></span> <span data-ttu-id="ffb83-131">次に、いくつかの基本原則に基づいてソリューションを設計しています。</span><span class="sxs-lookup"><span data-stu-id="ffb83-131">We then engineered a solution based on several guiding principles.</span></span>
