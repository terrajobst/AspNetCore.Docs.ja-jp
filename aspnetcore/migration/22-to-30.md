## <a name="update-net-core-sdk-version-in-globaljson"></a>global.json での .NET Core SDK バージョンの更新

ソリューションが特定の .NET Core SDK バージョンを対象とする[global.json](/dotnet/core/tools/global-json)ファイルに`version`依存している場合は、そのプロパティをコンピューターにインストールされている 3.0 バージョンに更新します。

```json
{
  "sdk": {
    "version": "3.0.100"
  }
}
```

### <a name="remove-obsolete-package-references"></a>古いパッケージ参照を削除する

```xml
    <TargetFramework>netcoreapp2.2</TargetFramework>

  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="Microsoft.AspNetCore.App"/>
    <PackageReference Include="Microsoft.AspNetCore.Razor.Design" Version="2.2.0" PrivateAssets="All" />
  </ItemGroup>

</Project>
```

### <a name="framework-reference"></a>フレームワークリファレンス

上記のパッケージのいずれかを通じて利用可能であったASP.NET Core の機能は、`Microsoft.AspNetCore.App`共有フレームワークの一部として利用できます。 *共有フレームワーク*は、コンピューター上にインストールされたアセンブリ (*.dll* ファイル) のセットであり、ランタイム コンポーネントと Targeting Pack を含みます。 詳しくは、[共有フレームワーク](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)に関するページをご覧ください。

* `Microsoft.NET.Sdk.Web` SDK を対象とするプロジェクトは、`Microsoft.AspNetCore.App` フレームワークを暗黙に参照します。

  これらのプロジェクトには、追加の参照は必要ありません。

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* `Microsoft.NET.Sdk`ターゲットまたは`Microsoft.NET.Sdk.Razor`SDK を対象とするプロジェクトは`FrameworkReference`、`Microsoft.AspNetCore.App`に明示的に追加する必要があります。

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>Docker を使用したフレームワーク依存型ビルド

core[共有フレームワーク](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/)ASP.NETに依存するパッケージを使用するコンソール アプリのフレームワーク依存ビルドでは、次のランタイム エラーが発生する可能性があります。

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`は、ASP.NETコア ランタイムを含む共有フレームワークであり、[ドットネット/コア/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker イメージ上にのみ存在します。 3.0 SDK では、共有フレームワークで使用可能なライブラリの重複コピーを含めないようにして、ASP.NET Core を使用してフレームワークに依存するビルドのサイズを削減します。 これは最大 18 MB の節約になる可能性がありますが、アプリを実行するにはASP.NET Core ランタイムが存在する/インストールする必要があります。

アプリがASP.NET Core 共有フレームワークに依存しているかどうか (直接的または間接的) を判断するには、アプリのビルド/公開中に生成された*runtimeconfig.json*ファイルを調べます。 次の JSON ファイルは、ASP.NETコア共有フレームワークへの依存関係を示しています。

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

アプリで Docker を使用している場合は、Core 3.0 ASP.NET含むベース イメージを使用します。 たとえば、「 `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0` 」のように入力します。

## <a name="kestrel"></a>Kestrel

### <a name="configuration"></a>構成

によって提供される Web ホスト ビルダーに Kestrel 構成を移行`ConfigureWebHostDefaults`する (*Program.cs*):

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

アプリがホストを手動で`HostBuilder`作成する場合は`UseKestrel`、 の Web`ConfigureWebHostDefaults`ホスト ビルダーを呼び出します。

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

    host.Run();
}
```

### <a name="connection-middleware-replaces-connection-adapters"></a>接続ミドルウェアが接続アダプタを置き換える

接続アダプタ (`Microsoft.AspNetCore.Server.Kestrel.Core.Adapter.Internal.IConnectionAdapter`) が Kestrel から削除されました。 接続アダプタを接続ミドルウェアに置き換えます。 接続ミドルウェアは、ASP.NETコアパイプラインの HTTP ミドルウェアに似ていますが、下位レベルの接続に対してはあります。 HTTPS と接続ログ:

* 接続アダプタから接続ミドルウェアに移動しました。
* これらの拡張メソッドは、以前のバージョンの ASP.NET Core と同様に機能します。 

詳細については、[ケストレルの記事の「リッスン オプション.プロトコル」セクションの TlsFilterConnectionHandler の例を](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#listenoptionsprotocols)参照してください。

### <a name="transport-abstractions-moved-and-made-public"></a>輸送の抽象化が移動し、公開

Kestrel トランスポート層は、`Connections.Abstractions` のパブリック インターフェイスとして公開されています。 これらの更新プログラムの一部として:

* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions`および関連付けられている型が削除されました。
* <xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions.NoDelay>は、トランスポート<xref:Microsoft.AspNetCore.Server.Kestrel.Core.ListenOptions>オプションに移動されました。
* `Microsoft.AspNetCore.Server.Kestrel.Transport.Abstractions.Internal.SchedulingMode`から<xref:Microsoft.AspNetCore.Server.Kestrel.KestrelServerOptions>削除されました。

詳細については、次の GitHub リソースを参照してください。

* [クライアント/サーバー ネットワークの抽象化 (ドットネット/AspNetCore #10308)](https://github.com/dotnet/AspNetCore/issues/10308)
* [新しい岩盤リスナーの抽象化を実装し、上にケストレルを再plat (ドットネット/AspNetCore #10321)](https://github.com/dotnet/AspNetCore/pull/10321)

### <a name="kestrel-request-trailer-headers"></a>ケストレルリクエストトレーラーヘッダー

以前のバージョンのASP.NET Core を対象とするアプリの場合:

* Kestrel は、HTTP/1.1 チャンク化トレーラー・ヘッダーを要求ヘッダー・コレクションに追加します。
* トレーラーは、要求本文が最後まで読み取られた後に使用できます。

これにより、ヘッダーとトレーラーの間のあいまいさが懸念されるため、トレーラーは 3.0 の新しいコレクション (`RequestTrailerExtensions`) に移動されました。

HTTP/2 リクエストトレーラーは次のとおりです。

* ASP.NETコア 2.2 では使用できません。
* 3.0 で使用`RequestTrailerExtensions`できます。

これらのトレーラーにアクセスするための新しい要求拡張メソッドが存在します。 HTTP/1.1 と同様に、要求本文が最後まで読み取られた後にトレーラーが使用可能になります。

3.0 リリースでは、以下`RequestTrailerExtensions`のメソッドを使用できます。

* `GetDeclaredTrailers`&ndash;本文の後`Trailer`にどのトレーラーが予期されるのかを一覧表示する要求ヘッダーを取得します。
* `SupportsTrailers`&ndash;要求がトレーラー ヘッダーの受信をサポートするかどうかを示します。
* `CheckTrailersAvailable`&ndash;リクエストがトレーラーをサポートしているかどうか、および読み取り可能かどうかを確認します。 このチェックでは、読み取るトレーラーがあるとは想定していません。 このメソッドによって返された場合`true`でも、読み取るトレーラーがない可能性があります。
* `GetTrailer`&ndash;要求された末尾ヘッダーを応答から取得します。 を`SupportsTrailers`呼び`GetTrailer`出す前<xref:System.NotSupportedException>にチェックするか、要求が末尾ヘッダーをサポートしていない場合は a が発生する可能性があります。

詳細については、「[要求トレーラーを別のコレクションに配置する (ドットネット/AspNetCore #10410)」](https://github.com/dotnet/AspNetCore/pull/10410)を参照してください。

### <a name="allowsynchronousio-disabled"></a>同期IO を無効にする

`AllowSynchronousIO``HttpRequest.Body.Read`などの同期 I/O API を`HttpResponse.Body.Write`有効または無効に`Stream.Flush`します。 これらの API は、アプリのクラッシュにつながるスレッドの枯し込みの原因です。 3.0 では、`AllowSynchronousIO` は既定で無効になっています。 詳細については[、Kestrel の記事の「同期 I/O」セクションを](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#synchronous-io)参照してください。

同期 I/O が必要な場合は、使用するサーバーで`AllowSynchronousIO`オプションを構成することによって有効にすることができます (呼`ConfigureKestrel`び出し時など、Kestrel を使用している場合)。 サーバー (Kestrel、HttpSys、テストサーバーなど) には、他のサーバーに影響`AllowSynchronousIO`を与えない独自のオプションがあることに注意してください。 以下のオプションを使用して、要求ごとにすべてのサーバーに対して同期 I/O`IHttpBodyControlFeature.AllowSynchronousIO`を有効にすることができます。

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

<xref:System.IO.TextWriter> [Dispose](/dotnet/standard/garbage-collection/implementing-dispose)で同期 API を呼び出す実装やその他のストリームで問題が発生<xref:System.IO.Stream.DisposeAsync*>した場合は、代わりに新しい API を呼び出します。

詳細については[、「[アナウンス] すべてのサーバーで同期IOを無効にする (ドットネット/AspNetCore #7644)」](https://github.com/dotnet/AspNetCore/issues/7644)を参照してください。

### <a name="security-middleware-guidance"></a>セキュリティミドルウェアのガイダンス

承認と CORS のサポートは、[ミドルウェア](xref:fundamentals/middleware/index)のアプローチを中心に統一されています。 これにより、これらのシナリオで同じミドルウェアと機能を使用できます。 このリリースでは、更新された認証ミドルウェアが提供され、CORS ミドルウェアが強化され、MVC コントローラーで使用される属性を理解できます。

#### <a name="cors"></a>CORS

以前は、CORS の設定が困難でした。 ミドルウェアは、いくつかのユースケースで使用するために提供されましたが、MVCフィルタは、他のユースケースではミドルウェア**なしで**使用することを意図していました。 ASP.NET Core 3.0 では、CORS を必要とするすべてのアプリがエンドポイント ルーティングと並行して CORS ミドルウェアを使用することをお勧めします。 `UseCors`は、デフォルト・ポリシーを提供することができ`[EnableCors]`、また`[DisableCors]`属性を使用して、必要に応じてデフォルト・ポリシーをオーバーライドできます。

次の例では

* CORS は、指定されたポリシーを持`default`つすべてのエンドポイントで有効になります。
* この`MyController`クラスは、属性を持つ`[DisableCors]`CORS を無効にします。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

アプリでカスタム[承認ハンドラー](xref:security/authorization/policies#authorization-handlers)を使用する場合、エンドポイント ルーティングは、MVC とは異なる種類のリソースをハンドラーに渡します。 承認ハンドラー コンテキスト リソースの種類<xref:Microsoft.AspNetCore.Mvc.Filters.AuthorizationFilterContext> [(MVC フィルターによって提供される](xref:security/authorization/policies#accessing-mvc-request-context-in-handlers)リソースの種類 ) を要求するハンドラーは、型<xref:Microsoft.AspNetCore.Routing.RouteEndpoint>のリソース (エンドポイント ルーティングによって承認ハンドラーに与えられるリソースの種類) を処理するために更新する必要があります。

MVC は`AuthorizationFilterContext`依然としてリソースを使用するため、アプリが MVC 承認フィルターとエンドポイント ルーティング承認を使用する場合は、両方の種類のリソースを処理する必要があります。

### <a name="mvc-controllers"></a>MVC コントローラ

コントローラのマッピングが内部`UseEndpoints`で行われるようになりました。

アプリ`MapControllers`が属性ルーティングを使用する場合は、追加します。 ルーティングには ASP.NET Core 3.0 以降の多くのフレームワークのサポートが含まれているため、属性ルーティング コントローラの追加はオプトインです。

次の部分を探します。

* `MapControllerRoute` を含む `MapRoute`
* `MapAreaControllerRoute` を含む `MapAreaRoute`

ルーティングには MVC 以上のサポートが含まれているため、これらのメソッドが何を行うかを明確に示すために用語が変更されました。 `MapControllerRoute`/などの`MapAreaControllerRoute`従来/ルートは、追加された順序で適用`MapDefaultControllerRoute`されます。 より具体的なルート(エリアのルートなど)を最初に配置します。

次の例では

* `MapControllers`属性ルーティング コントローラのサポートが追加されました。
* `MapAreaControllerRoute`エリア内のコントローラの従来のルートを追加します。
* `MapControllerRoute`コントローラの従来のルートを追加します。

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
        endpoints.MapAreaControllerRoute(
            "admin",
            "admin",
            "Admin/{controller=Home}/{action=Index}/{id?}");
        endpoints.MapControllerRoute(
            "default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

#### <a name="async-suffix-removal-from-controller-action-names"></a>コントローラのアクション名からの非同期サフィックスの削除

core 3.0 ASP.NETでは、core MVC `Async` ASP.NETコントローラー アクション名からサフィックスを削除します。 ルーティングとリンクの生成は、この新しいデフォルトの影響を受けます。 次に例を示します。

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

core 3.0 ASP.NET前のバージョン:

* 上記のアクションは *、製品/リスト非同期*ルートでアクセスできます。
* リンクの生成には、サフィックス`Async`を指定する必要があります。 次に例を示します。

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

ASP.NETコア 3.0 で:

* 前のアクションは *、製品/リスト*ルートでアクセスできます。
* リンクの生成では、サフィックスを指定する`Async`必要はありません。 次に例を示します。

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

この変更は、 属性を使用して指定[`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute)された名前には影響しません。 既定の動作は、 の次のコードで`Startup.ConfigureServices`無効にすることができます。

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>リンク生成の変更

[以前のバージョンのルーティングとの違いに](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)関するドキュメントで説明したように、リンク生成にはいくつかの違`Url.Link`いがあります(たとえば、APIの使用や類似の API など)。 次の設定があります。

* デフォルトでは、エンドポイントルーティングを使用する場合、生成された URI のルート パラメータの大文字と小文字は必ずしも保持されません。 この動作は`IOutboundParameterTransformer`、インターフェイスで制御できます。
* 無効なルート (コントローラー/アクション、または存在しないページ) の URI を生成すると、無効な URI を生成する代わりに、エンドポイント ルーティングの下に空の文字列が生成されます。
* アンビエント値(現在のコンテキストからのルート パラメータ)は、エンドポイント ルーティングとのリンク生成で自動的には使用されません。 以前は、別のアクション(またはページ)へのリンクを生成する場合、指定されていないルート値は*現在*のルートのアンビエント値から推論されます。 エンドポイント ルーティングを使用する場合、リンク生成時にすべてのルート パラメータを明示的に指定する必要があります。

## <a name="opt-in-to-runtime-compilation"></a>ランタイム コンパイルにオプトインする

Core 3.0 ASP.NET以前は、ビューの実行時コンパイルはフレームワークの暗黙的な機能でした。 ランタイム コンパイルは、ビューのビルド時コンパイルを補足します。 これにより、アプリケーション全体を再構築することなく、ファイルが変更されたときに、フレームワークで Razor ビューとページ *(.cshtml*ファイル) をコンパイルできます。 この機能は、IDE でクイック編集を行い、ブラウザーを更新して変更を表示するというシナリオをサポートしています。

ASP.NET Core 3.0 では、ランタイム コンパイルはオプトイン シナリオです。 ビルド時コンパイルは、ビューのコンパイルのメカニズムとして、既定で有効になっている唯一のメカニズムです。 ランタイムは、Visual Studio または Visual Studio コードの[ドットネット ウォッチ](xref:tutorials/dotnet-watch)に依存して *、.cshtml*ファイルへの変更を検出したときにプロジェクトを再構築します。 Visual Studio で、実行されているプロジェクト内の *.cs、.cshtml、**または .razor*ファイルに変更を加え、デバッグされていない<kbd>(F5)</kbd>すると、プロジェクトの再コンパイルがトリガーされます。 *.cs*<kbd>Ctrl+F5</kbd>

ASP.NET Core 3.0 プロジェクトでランタイム コンパイルを有効にするには、次の手順を実行します。

1. パッケージ[を](https://www.nuget.org/packages/Microsoft.AspNetCore.Mvc.Razor.RuntimeCompilation)インストールします。
1. 呼`Startup.ConfigureServices`び出`AddRazorRuntimeCompilation`しに更新:

    ASP.NETコア MVC では、次のコードを使用します。

    ```csharp
    services.AddControllersWithViews()
        .AddRazorRuntimeCompilation(...);
    ```
 
    ASP.NETコア Razor ページの場合は、次のコードを使用します。
 
    ```csharp
    services.AddRazorPages()
        .AddRazorRuntimeCompilation(...);
    ```
 
のサンプルはhttps://github.com/aspnet/samples/tree/master/samples/aspnetcore/mvc/runtimecompilation、開発環境で条件付きでランタイム コンパイルを有効にする例を示しています。

Razor ファイルのコンパイルの詳細については、「」を参照してください<xref:mvc/views/view-compilation>。
